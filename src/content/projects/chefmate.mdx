## 背景与动机

ChefMate 不是为了解决"怎么做菜"这个需求——菜谱网站已经够多了。它是我用来自学 RAG 技术栈的 **工程化练习项目**：

- 理解 RAG 的完整工作流：文档加载 → 分块 → 向量化 → 检索 → 生成
- 实践混合检索策略（向量 + BM25）和 RRF 融合排序
- 体验 LLM 在 RAG 中的两个关键角色：查询路由 + 查询改写
- 学习如何将 CLI 工具封装为 Web 服务并部署上线

**核心设计哲学**：小块检索 → 大块生成。用子块精确命中用户问题，用父文档保证 LLM 上下文完整性。

## 技术选型

| 层级 | 技术 | 用途 |
|------|------|------|
| 向量数据库 | FAISS (384维, bge-small-zh-v1.5) | 语义检索 |
| 关键词检索 | BM25 (rank-bm25) | 关键词匹配 |
| 融合排序 | RRF (Reciprocal Rank Fusion) | 混合排序 |
| 嵌入模型 | BAAI/bge-small-zh-v1.5 | 文本向量化 |
| LLM | DeepSeek V4 (deepseek-chat) | 路由 + 改写 + 生成 |
| API 框架 | FastAPI | Web 服务 |
| 前端 | Vue3 CDN 单页 | 零构建 SPA |
| 部署 | Docker + Nginx | 容器化 |

**数据源**：712 个 Markdown 菜谱，按标题层级（H1/H2/H3）切分为约 2000 个子块，保留父子文档映射。

## 系统架构

```
用户输入
  │
  ├──► 查询路由 (LLM 分类: list/detail/general)
  │
  ├──► 查询改写 (Query Rewriting)
  │
  ├──► 混合检索
  │     ├── 向量检索 (FAISS, 语义匹配)
  │     └── BM25 检索 (关键词匹配)
  │     └── RRF 融合排序
  │
  ├──► 父文档替换 (子块 → 完整父文档)
  │
  └──► LLM 生成 (DeepSeek V4, 三种 Prompt)
```

### 关键设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 分块策略 | MarkdownHeaderTextSplitter | 按标题层级切分，不破坏语义 |
| 父子文档 | 子块检索 → 父文档生成 | 小块保精度，大块保召回 |
| 混合检索 | 向量 + BM25 + RRF | 语义互补关键词，RRF 消除分数尺度差异 |
| 流式输出 | SSE (Server-Sent Events) | 比 WebSocket 轻量，FastAPI 原生支持 |
| API 封装 | FastAPI | Python 原生异步，自动 OpenAPI 文档 |

## 核心模块详解

### 1. 数据准备：文档加载与分块

```python
# loader.py 核心逻辑
headers_to_split_on = [
    ("#", "主标题"),
    ("##", "二级标题"),
    ("###", "三级标题"),
]
markdown_splitter = MarkdownHeaderTextSplitter(
    headers_to_split_on=headers_to_split_on,
    strip_headers=False,  # 保留标题，增强上下文
)
```

每个子块保留 `parent_id` 和 `chunk_id`，通过 `parent_child_map` 维护父子关系。检索返回子块后，替换为完整的父文档（含所有子块）送入 LLM。

### 2. 向量索引：嵌入 + FAISS

使用 `BAAI/bge-small-zh-v1.5`（384 维）将文本转为向量，FAISS 构建索引并持久化：

```python
# indexer.py
embeddings = HuggingFaceEmbeddings(
    model_name="BAAI/bge-small-zh-v1.5",
    model_kwargs={"device": "cpu"},
    encode_kwargs={"normalize_embeddings": True},
)
vectorstore = FAISS.from_texts(texts, embedding=embeddings, metadatas=metadatas)
vectorstore.save_local("vector_index/")
```

首次构建后，后续启动秒级加载。

### 3. 混合检索：向量 + BM25 + RRF

```python
# retriever.py
def hybrid_search(self, query, top_k=3):
    vector_docs = self.vector_retriever.invoke(query)     # 语义
    bm25_docs = self.bm25_retriever.invoke(query)         # 关键词
    return self._rrf_rerank(vector_docs, bm25_docs)[:top_k]

def _rrf_rerank(self, vec_results, bm25_results, k=60):
    rrf_scores = {}
    for rank, doc in enumerate(vec_results):
        rrf_scores[id(doc)] += 1 / (k + rank + 1)
    for rank, doc in enumerate(bm25_results):
        rrf_scores[id(doc)] += 1 / (k + rank + 1)
    # 按 RRF 分数降序排列
```

**为什么用 RRF？** 向量检索和 BM25 的分数尺度不同，无法直接相加。RRF 使用排名而非原始分数，消除了尺度差异。

### 5. RAG 指标计算

每次问答计算并返回以下指标：

| 指标 | 计算方式 | 用途 |
|------|---------|------|
| 置信度 | top_score × 0.5 + mean_score × 0.3 + 来源覆盖 × 0.4 | 回答可信度 |
| 来源证据 | 每个匹配的父文档：菜名、分类、相关性分数 | 检索透明度 |
| 检索耗时 | 端到端计时 | 性能监控 |
| 路由类型 | LLM 分类结果 | 行为可解释 |

## API 设计

| 方法 | 路径 | 说明 | 限流 |
|------|------|------|------|
| `POST` | `/api/chat` | 非流式问答 | 10/min/IP |
| `POST` | `/api/chat/stream` | SSE 流式问答 | 5/min/IP |
| `GET` | `/api/history` | 对话历史 | 30/min/IP |
| `POST` | `/api/feedback` | 用户反馈 | 20/min/IP |
| `GET` | `/api/health` | 健康检查 | 不限 |

核心请求/响应格式：

```json
// POST /api/chat
// Request:  {"query": "宫保鸡丁怎么做"}
// Response:

  "answer": "# 宫保鸡丁制作指南\n\n...",

    "sources": [

    ],
    "elapsed_ms": 2141.8,
    "query": {"route_type": "detail"}
  }
}
```

## 安全防护与成本控制

DeepSeek API 按 token 付费，五层防线防止恶意刷量：

| 防线 | 措施 | 配置 |
|------|------|------|
| 频率限制 | slowapi 每 IP 限流 | 10/min（chat），5/min（stream） |
| 每日配额 | SQLite 计数 + 环境变量 | DAILY_QUOTA=200 |
| 输入校验 | Pydantic 拒绝 HTML/超长/重复 | max_length=500 |
| 异常检测 | api_logs 表 + 定时检查 | 同一 IP > 50次/h → 封禁 |
| Token 预算 | max_tokens=2048 | 单次约 ￥0.004 |

**成本预估**：每日 50 次约 ￥0.2，月费 < ￥10。

> API Key 存储在服务器端 `.env`，前端永远拿不到。所有请求经由 FastAPI 中转，不直接暴露 DeepSeek。

## 前端集成

ChefMate 以子路径 `/chefmate/` 集成在个人网站内，Nginx 将 `/api/*` 代理到 FastAPI 后端：

```
vincentbuilds.fun
  ├── /              → Astro 静态站 (首页/博客/项目)
  ├── /chefmate/     → ChefMate 聊天页面
  └── /api/*         → Nginx 反向代理 → FastAPI :8000
```

这样做的好处：
- **单一域名**：不需要额外购买域名
- **共享 SSL**：已有 HTTPS 证书直接复用
- **统一导航**：ChefMate 作为网站的一个页面
- **无需 CORS**：同源请求，省去跨域配置
